# SPEC.md — pharos

## §G goal

Rust media server. Wire-compat with Jellyfin clients (Phase 1) and Plex clients (Phase 2). Better perf/reliability than both. Group watch + group listen first-class, not bolt-on. Group-sync is a **primary motivation** — Jellyfin's SyncPlay is buggy in practice (late-joiner desync, poor-network member drags whole group, buffer storms on leader handoff). pharos must improve on those, not replicate them.

**Phase 1 priority order**: (1) Jellyfin client compat, (2) Dioxus web UI, (3) core (scan/store/group-sync). Plex (T11–T14) waits until Phase 1 is solid.

## §C constraints

- Lang: Rust stable. Backend + frontend both Rust.
- HTTP framework: actix-web (perf over axum).
- CLI: clap derive (declarative) mode. No builder API.
- DB: sqlx. All DB access behind `pharos-core` traits (e.g. `MediaStore`). Backend impls (sqlite default, postgres optional) live in adapter crate. Call sites depend on traits only — swap via wiring, not refactor.
- Async traits: native `async fn` in traits (stable Rust 1.75+). No `async_trait` crate. Prefer generics over `dyn Trait` for swappable backends.
- SIMD: use SIMD-accelerated crates where applicable. JSON parsing via `sonic-rs` (ByteDance; 1.5–2× faster than `simd-json` on common workloads, no intermediate tape). Hashing via `xxhash-rust` (xxh3) for stable IDs and `blake3` for content hashing. Image decode via `image` (default SIMD). Fall back gracefully on non-supported archs.
- Benchmarks: every SIMD-accelerated or otherwise hot code path has a `criterion` benchmark under the owning crate's `benches/`. Bench gates regressions, not unit correctness.
- Test runner: `cargo nextest run --workspace`. Config in `.config/nextest.toml`. Doctests via `cargo test --doc` separately.
- Web UI: Dioxus (Rust → WASM). Single-lang stack.
- API compat: existing Jellyfin clients (Finamp, Infuse, Jellyfin web/mobile/TV) work unmodified — Phase 1 hard requirement.
- API compat: existing Plex clients (Plex web/mobile/TV, Plexamp) work unmodified — **Phase 2; deprioritised**. T11–T14 hold but no Plex work blocks Jellyfin progress.
- Phase 1: feature-parity with Jellyfin + pharos-native group-sync.
- Phase 2: Plex client compat + extensions past Jellyfin parity.
- Transcoding via ffmpeg (subprocess, no FFI initially).
- Single binary deploy. Config file + env vars. No external runtime deps beyond ffmpeg.
- TDD: failing test before impl change. No "test later".
- Design: mine Jellyfin structure/feature list for ideas. Adapt idiomatic Rust — traits at IO/abstraction boundaries, not OOP class translation.
- Observability baked from T1. Not retrofit. Traces + metrics + structured logs always-on.
- Concurrency: actor pattern (tokio task + mpsc) for mutable runtime state. Locks only for one-shot init or proven read-only caches.
- Dev env: Nix flake canonical. `nix develop` → reproducible shell with rust toolchain, clippy, rustfmt, ffmpeg. CI uses same flake.
- Build artifacts: `nix build .#pharos` → server binary. `nix build .#oci` → OCI image via `dockerTools`. No `Dockerfile`.

## §I interfaces

- jellyfin-api — HTTP/REST surface matching Jellyfin schemas (auth, system, library, items, playback, sessions).
- plex-api — HTTP/XML+JSON surface matching Plex schemas (auth, library, hubs, streaming, sessions).
- group-sync — websocket protocol for synced playback across clients in shared session.
- media-fs — filesystem scan target. Configured roots. Watch for changes.
- ffmpeg — subprocess for transcode/probe/thumbnail.
- store — sqlite (default) or postgres for metadata/users/sessions.
- config — TOML file + env override. Path via `--config` or `PHAROS_CONFIG`.
- cli — `pharos serve`, `pharos scan`, `pharos admin <subcommand>`.
- dioxus-ui — Dioxus web frontend, served by backend. WASM bundle. Replaces Jellyfin-web role.
- obs — OpenTelemetry traces (OTLP exporter) + Prometheus metrics (`/metrics`) + structured logs via `tracing` crate.
- health-api — `/healthz` liveness, `/readyz` readiness, `/info` build+version.
- nix-flake — `nix develop` devShell; `nix build .#pharos` server; `nix build .#oci` OCI image (dockerTools).

## §V invariants

- V1: unmodified Jellyfin client connects, browses library, plays media (direct + transcoded).
- V2: unmodified Plex client connects, browses library, plays media (direct + transcoded).
- V3: group session syncs play/pause/seek across members within 500ms p95.
- V4: HTTP handler never panics. Errors return structured response matching target API schema.
- V5: library scan never blocks playback or API requests. Scan runs in dedicated task pool.
- V6: ffmpeg subprocess crash never crashes server. Failed transcode returns error to client.
- V7: API response shapes match Jellyfin/Plex reference schemas byte-equivalent for shared fields.
- V8: auth tokens never logged. Secrets redacted in all log output.
- V9: media file path never leaks to unauthenticated client. Path traversal blocked at boundary.
- V10: store writes atomic per logical op. No partial library entry visible to readers.
- V11: every public behavior covered by test written before impl (TDD). No merge without failing-test-first evidence.
- V12: trait boundary between domain logic and IO. Domain layer testable without DB/fs/network/ffmpeg.
- V13: every inbound request traced. Every outbound IO (db, ffmpeg, fs walk, http client) spanned. trace_id propagated to logs.
- V14: `/healthz` returns 200 if process alive. `/readyz` returns 200 only when store reachable + scanner initialized. `/metrics` exposes Prometheus format.
- V15: structured logs only. No `println!`/`eprintln!` in non-test code. `tracing` crate sole logging surface. CLI subcommands may write to stdout/stderr via explicit `std::io::Write` (not logging).
- V16: Dioxus UI consumes only public Jellyfin-compat API. No backdoor endpoints. UI swappable for any Jellyfin client.
- V17: no `unwrap()` / `expect()` in non-test code. Enforced via `clippy::unwrap_used` + `clippy::expect_used` = deny at workspace level. Tests may opt-out.
- V18: mutable runtime state owned by exactly one task. Mutation via message-passing (tokio mpsc). No `Mutex<State>` on request path. Locks permitted only for one-shot init or immutable-after-init caches.
- V19: group-sync must improve on Jellyfin SyncPlay failure modes. Specifically: (a) late joiner does not desync existing members; (b) one poor-network member does not stall the group (per-member buffering isolation); (c) leader handoff causes ≤1 corrective Pause, no buffer storm; (d) network blip <2 s reconverges automatically without member rejoin.
- V20: group-sync wire protocol stays Jellyfin SyncPlay-compatible so unmodified phone/TV clients participate. Improvements live in server-side algorithm. A second WS path may carry an extended protocol for pharos-native clients, but the Jellyfin-shaped socket is the canonical entry point.
- V21: SyncPlay server never withholds command client must ACK. Readiness gate waits only on members expected to emit player transition (queue load, already-delivered Seek, active buffering). Plain Unpause of non-buffering group broadcasts immediately. Every SendCommand carries PositionTicks (jellyfin-web seeks to it — missing value = seek to 0). SetIgnoreWait honored: halted member excluded from all gates.
- V22: Quick Connect wire flow matches jellyfin-web two-endpoint exchange. `/QuickConnect/Connect` is read-only poll — echoes `Secret`, reports `Authenticated`, never mints token, never consumes. Finalize at `POST /Users/AuthenticateWithQuickConnect` {Secret} (unauth — secret IS credential): single-shot consume of an AUTHORIZED entry → AuthenticationResult (User + AccessToken). Unauthorized/unknown/spent secret → 401. Its query params (`secret`, `code`) bind case-INSENSITIVELY — mobile/TV SDKs send camelCase, jellyfin-web PascalCase; real Jellyfin (ASP.NET) is case-insensitive, so a case-sensitive extractor 400s non-web clients.
- V23: software video transcode jobs are thread-bounded + speed-preset. Every sw encode AND decode capped at `sw_encode_threads()` (≈4); libx264/libx265 get `-preset veryfast` (never x264 default `medium` — below realtime under load); CPU permit budget = `cores / sw_encode_threads()`, never permits=cores (double-counts: each job auto-threads all cores, all admitted jobs slow below realtime together); prefetch depth derives from permits, capped at 4. Invariant: `permits × threads-per-job ≤ cores`.
- V24: a synth group id (Series/Season/Artist/Album) serves LEVEL-specific art before any representative-member fallback. Seasons: probe the Kodi/Jellyfin series-root sidecars (`season{NN}-poster/-fanart/-banner`, zero-padded + bare, `season-specials-*` for S0) for the season the id names; only then fall back to the representative episode's artwork (series poster). Fallback must never make two different groups of one show serve identical bytes when level-specific art exists on disk.
- V32: a member's delivery sink in `MemberSinks` is GENERATION-FENCED — every insert/remove carries the owning socket's `conn_gen`. An insert lands only when its gen ≥ the stored gen; a remove fires only when its gen == the stored gen. So no stale writer (the losing task of two racing connects for one deterministic member_id, or an HTTP New/Join re-registering a sink snapshot resolved just before a reconnect) can ever install a dead sink over the live one, and no late disconnect teardown can wipe the sink a fresh reconnect just registered. A blackholed sink = the member silently receives NO SyncPlay traffic until its next reconnect (the "got stuck / never started" failure). The Jellyfin `/socket` path uses the hub's per-device conn_gen; the native `/sync/v1/ws` path uses a process-wide monotonic counter.
- V31: the V19 per-member buffering freeze is BOUNDED — one member's `BufferingStart` may pause the group, but the group ALWAYS resumes. It resumes (a) the moment the last buffering member reports `BufferingEnd`/`Ready`, (b) immediately when the last buffering member LEAVES (socket drop / Leave / ghost-prune) without a `BufferingEnd` — a departing member must never own the freeze — or (c) after `BUFFERING_MAX_MS` (anti-wedge) if a member buffers forever without recovering, ending, or leaving. A group hydrated mid-freeze (rolling deploy) re-arms the anti-wedge deadline, since its restored members carry cleared buffering flags and can never fire the `BufferingEnd` auto-resume on their own. Mirror of V28's readiness-gate anti-wedge, for the buffering path.
- V25: a SyncPlay watch party survives a process restart / rolling deploy end-to-end, and the server NEVER silently drops a group command. Member ids derive deterministically from deviceId (same device → same member id across restarts); on /socket connect with no in-memory membership the server recovers it from the persisted `sync_groups` snapshots and resyncs the client; a command from a session with no recoverable group answers `NotInGroup` over the socket (stock jellyfin-web then visibly disables SyncPlay) — a silent drop lets the sender's local player apply the command one-sidedly (Pause pauses only the presser) and desyncs the party.
- V29: a trickplay sprite set counts as GENERATED only when its per-(media,width) `.complete` marker exists — written strictly after every sheet landed. Tile presence alone is a lie: the libav worker writes sheets straight into the served dir, so an interrupted run (op timeout, worker crash, deploy) leaves a truncated set that must be re-attempted, never advertised (grey previews past the truncation point otherwise). Marker-less sets are verified against the duration-derived expected sheet count (±1 trailing sheet for VFR/rounded durations): consistent → stamp marker; truncated → wipe + regenerate. Whole-file heavy-op timeouts scale with content duration (`max(900s, duration/2)`) — a fixed cap kills long movies partway. Image-subtitle burn-in uses the `overlay` filter_complex (the `subtitles=` filter is text-only and cannot render PGS/VOBSUB).
- V28: SyncPlay command scheduling is BOUNDED and gate ACKs are causally valid. (a) Clock samples with RTT > 10s are discarded at observe time (a throttled/suspended tab measures its own stall, not the network — its offset estimate is garbage too); (b) `lead_time_ms = MIN_LEAD + min(max_rtt/2, 2s)` — one member's pathological RTT must never schedule commands tens of seconds out for the whole group; (c) a readiness gate opened for a SCHEDULED command (Seek: clients execute at `When = at_server_ms`) ignores any `Ready` arriving before `at_server_ms` — it cannot be an ACK of a command nobody has executed yet; it's a spurious player transition. Counting it resolves the gate early, and the resulting Play broadcast makes jellyfin-web's `scheduleUnpause` CANCEL the client's still-pending seek callback (`clearScheduledCommand`), after which nothing ever seeks — `scheduleUnpause` only corrects players AHEAD of target, so everyone silently resumes from the pre-seek position.
- V27: every client-attributed SyncPlay report or request that names a queue entry (`Ready`, `Buffering`, `NextItem`, `PreviousItem` — all carry `PlaylistItemId`) is validated against the CURRENT queue entry before it acts. A stale-entry `Ready`/`Buffering` (the old episode's player transitioning during teardown, racing a queue change) must neither satisfy a readiness gate nor freeze the group — the sender gets a single-member PlayQueue catch-up re-send (same `LastUpdate`, so up-to-date clients drop it as a duplicate) and is expected to re-report for the right entry. A stale-entry `NextItem`/`PreviousItem` is a no-op (double-press / two-member race would skip an episode). jellyfin-web posts `Ready` on EVERY player transition, so an unvalidated gate is trivially poisoned by teardown transitions. Wire `UserLeft` carries the member's display NAME, never the member uuid.
- V30: the segmented-HLS surface (`/hls1/*.ts`) NEVER stream-copies video and never passes multichannel audio. `-c:v copy` per-segment HLS is structurally broken three ways: ffmpeg cuts copy output on source keyframes (durations drift off the uniform EXTINF grid), `-output_ts_offset` is silently INERT under copy (every segment restarts at PTS≈0), and passthrough/re-encoded multichannel AAC is undecodable in Firefox's MSE (fatal append on segment 0 — playback never starts; Chrome tolerates all three, masking the breakage). Every `.ts` segment: re-encoded H.264, `-muxdelay 0 -muxpreload 0` (default 1.4 s mpegts cue delay skews the anchor), `-output_ts_offset <start>`, AAC forced `-ac 2`. Copy remux stays legal ONLY on the progressive `/stream` path (one continuous output, no per-segment cuts). Corollary: any ffmpeg flag doing timeline/mux work needs a BEHAVIORAL test (run ffmpeg, ffprobe the output) — an argv-string assertion passes even when the muxer ignores the flag (B41 shipped exactly that way).
- V31: image-subtitle burn-in pays ONLY for segments the track provably subtitles. Every burn-capable request (live .ts, VP9 init/media, both prefetch paths) gates against the track's scanned event-window timeline (TinyOp::SubtitleWindows — packet demux, no decode; persisted per file+mtime+track via SubtitleKind::EventWindows); an event-free segment window (±500ms pad) strips the burn index, takes the plain fast path, and shares its cache key with subtitle-off playback. Gating is FAIL-OPEN: unknown/in-flight/failed timeline → burn conservatively — it may only remove provably-empty burns, never lose a visible subtitle. A nonexistent track index errors, never returns an empty (= never-burn) timeline. Prefetch re-gates per TARGET segment (the requesting segment's decision does not transfer on a sparse track). Any segment transcoding >3× its content duration logs WARN with full attribution (queue-wait/encode split, device, codec, burn, seek depth, source).
- V26: every server-side UserData mutation (mark played/unplayed, favorite/unfavorite, progress save, stop-finish) broadcasts a `UserDataChanged` frame whose `UserDataList` entries are FULL serialized `UserItemDataDto`s — `ItemId` in canonical 32-hex wire form, `Key`, `Played`, `IsFavorite`, `PlayCount`, `PlaybackPositionTicks`, `PlayedPercentage` (computed from runtime when known). jellyfin-web patches cards by `data-id == ItemId` and the detail page by `Key`, then applies the state fields in place — a bare `{ItemId}` stub or a decimal id matches nothing, so the UI only updates on manual refresh. Synthetic series/season ids cascade the mutation to every child episode and batch all changed entries (+ a folder entry keyed by the synth id) into ONE frame.
